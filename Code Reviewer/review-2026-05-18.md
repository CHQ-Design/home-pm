## Must Fix

### 1. `parseReminder` accepts `0` — and `0` is almost certainly wrong

**Issue:** `lib/parse.ts` returns `0` as a valid reminder value because `0 >= 0`. This means a user can set a reminder for "0 minutes before", which fires the cron immediately at due-time but will also trip the cron's `windowStart` guard in a way that makes reminders flaky.

**Why it matters:** `0` is not a meaningful reminder. It should either be treated as "no reminder" or given a minimum of 1. As-is, a form that submits an empty reminder string could potentially coerce to `0` depending on parsing path.

**Minimal fix:**
```ts
// lib/parse.ts
export function parseReminder(raw: string | null | undefined): number | null {
  if (!raw || raw === "") return null
  const n = parseInt(raw, 10)
  return isNaN(n) || n <= 0 ? null : n  // changed: n < 0 → n <= 0
}
```

### 2. `addNote` accepts `projectIdRaw` without integer validation before `Number()`

**Issue:** In `notes/actions.ts`, the `projectIdRaw` value is passed directly to `Number()` without going through the safe `parseId()` helper that other actions use.

```ts
// current — risky
const projectId = projectIdRaw ? await verifyBelongsToHousehold("project", Number(projectIdRaw), sessionUser.householdId) : null
```

`Number("1abc")` → `NaN`, which Prisma will reject with a runtime error instead of a clean null. `verifyBelongsToHousehold` doesn't guard against `NaN`. The other actions have a shared `parseId` helper for exactly this.

**Minimal fix:**
```ts
// notes/actions.ts — add parseId (or import from a shared location)
function parseId(raw: string | null): number | null {
  if (!raw) return null
  const n = parseInt(raw, 10)
  return isNaN(n) || n <= 0 ? null : n
}

// then in addNote:
const projectId = projectIdRaw
  ? await verifyBelongsToHousehold("project", parseId(projectIdRaw), sessionUser.householdId)
  : null
```

Better yet — consolidate `parseId` into `lib/parse.ts` and import it everywhere. Right now it's copy-pasted into `actions.ts` and `recurring/actions.ts` separately.

### 3. Duplicate `inputClass` definition in notes components

**Issue:** `app/notes/note-item.tsx` and `app/notes/add-note-form.tsx` both define their own local `const inputClass = "..."` string instead of importing from `lib/styles.ts`. The strings are close but not identical to the shared one (slightly different font sizes), meaning visual inconsistency will creep in over time.

**Why it matters:** The shared `lib/styles.ts` exists for exactly this purpose. This is scope creep in the other direction — not adding complexity, but duplicating what already exists.

**Fix:** Delete the local definitions in both files and import from `@/lib/styles`.

### 4. Cron route fetches all tasks/routines for every household, then filters nothing cross-household

**Issue:** The `/api/cron/notify` route queries all `task` and `recurringTask` records across the entire database with no `householdId` filter. It then falls back to `adminEmail` (the first admin found anywhere) for tasks with no assignee email.

```ts
const adminUser = await prisma.user.findFirst({ where: { role: "admin" } })
```

This is a cross-household data leak waiting to happen. If a task has no assignee email, it gets routed to a random admin across any household. For a single-household deploy this is currently harmless — but it's wrong.

**Minimal fix:** The cron doesn't need to be per-household yet. The safe patch is just to stop falling back to a random admin for unassigned tasks. Send to no-one if there's no assignee email:

```ts
const email = task.assignee?.email?.toLowerCase()
if (!email) continue  // skip — no one to notify
```

And remove the `adminUser` lookup entirely. If you want to notify admins for unassigned tasks in the future, that can be an intentional per-household feature — not a silent fallback.

## Security

**Important — `/api/uploads/[filename]` leaks attachment existence cross-household.**

The route checks that the user is authenticated, and it checks that the `filename` is in the `attachment` table. But it does not check that the attachment's note belongs to the requesting user's household.

```ts
const att = await prisma.attachment.findFirst({
  where: { filename },
  select: { id: true, originalName: true, blobUrl: true },
})
```

Any authenticated user from household A can download an attachment from household B if they know the UUID filename. Because the filenames are UUIDs this is low probability in practice, but it is a real authorization gap.

**Fix:**
```ts
const sessionUser = await getSessionUser()
if (!sessionUser) return NextResponse.json({ error: "Unauthorized" }, { status: 401 })

const att = await prisma.attachment.findFirst({
  where: { filename },
  include: { note: { select: { householdId: true } } },
})
if (!att || att.note.householdId !== sessionUser.householdId) {
  return NextResponse.json({ error: "Not found" }, { status: 404 })
}
```

**Watch later — JWT token carries stale role/householdId.** The `jwt` callback in `lib/auth.ts` writes `role` and `householdId` into the token on every JWT refresh. But JWT tokens are not invalidated server-side when a user's role changes (e.g. admin demotes them). There's a window of up to the session duration where a demoted member can still act as admin. This is acceptable for a personal app where you control all the accounts, but document it — and note that `getSessionUser()` always re-queries the DB, which is the right pattern and mitigates this for server actions. The gap is only in places that trust the session object directly rather than re-querying.

**No meaningful security concerns** found in: server action scoping, input validation, CSP headers, file upload MIME/extension handling, calendar route scoping, superadmin gate, or subscription endpoint.

## Simplify

**`parseId` is copy-pasted in three places.** Move it to `lib/parse.ts` alongside `parseReminder`. Same for `TIME_RE` and `parseTime` — both are duplicated verbatim in `app/actions.ts` and `app/recurring/actions.ts`. Consolidate:

```ts
// lib/parse.ts
export function parseId(raw: string | null | undefined): number | null { ... }
export const TIME_RE = /^\d{2}:\d{2}$/
export function parseTime(raw: string | null | undefined): string | null { ... }
```

**`localDateString()` in `recurring-task-item.tsx`** reimplements logic that already exists in `lib/dates.ts`. Check `utcDateStr` — it may already do what you need, or a one-liner addition to dates.ts is the right place.

**The cron route's `computeNotifyAt` function** is duplicated between `api/cron/notify/route.ts` and potentially wherever reminders are set up. If this function lives in only one place it's fine, but watch it.

**`formatDate` is defined locally in `task-item.tsx`, `recurring-task-item.tsx`, and `notes/note-item.tsx`.** All three do the same thing. Move to `lib/dates.ts` once.

## Best-Practice Notes

**`toggleTask` streak logic uses wall-clock math for "yesterday".** `new Date(Date.now() - 86400000)` is fragile around DST boundaries and doesn't respect the user's timezone. The streak logic is already using UTC strings so it's probably fine in practice, but it's worth noting. Alternatively, `todayUTC()` is already a util — just derive "yesterday" from that string rather than epoch arithmetic.

**`addNote` and `updateNote` don't emit a consistent shape on error.** Most other actions `return { error: "..." }` on soft failures, but the notes actions `throw new Error(...)` on auth failure. This is consistent with the pattern across actions, but it means the client gets an unhandled Next.js server action error bubble rather than a `result.error` message. Consider making the pattern consistent: throw on auth/unexpected errors, return `{ error }` on user-correctable ones.

**The `computeNextDue` function in `recurring/actions.ts` uses local `Date` methods** (`setDate`, `setMonth`, etc.) which are timezone-aware and will behave differently based on the server's timezone. For a purely UTC Neon database, this can cause off-by-one-day bugs when the server is in a non-UTC zone (Vercel runs UTC, so currently safe, but fragile). Worth a comment.

**`web-push.ts` throws at module load time** if VAPID env vars are missing. This means any environment without VAPID set (e.g. a local dev clone without `.env`) will crash the entire app on first import. Consider a lazy initialization or a clearer startup error that doesn't break unrelated routes.

**The `role` field on `User` is a plain `String` in the schema** rather than a Postgres enum or `@db.VarChar` with a check constraint. The application code validates it correctly, but the database allows any string. Low risk since it's only written from server actions, but worth noting as a data integrity gap.
