## 🔴 Real Bugs (Fix These)

### 1. "At the time" reminder is silently discarded

**File:** `lib/parse.ts`, `app/actions.ts`, `app/recurring/actions.ts`

`parseReminder` rejects any value `<= 0`:

```ts
return isNaN(n) || n <= 0 ? null : n
```

But the UI offers "At the time" with `value: "0"`. When a user selects it, the server receives `"0"`, `parseReminder` returns `null`, and the reminder is stored as no reminder at all — silently. The user thinks they've set a reminder and they won't get one.

**Fix:** Change `<= 0` to `< 0`:

```ts
return isNaN(n) || n < 0 ? null : n
```

Also update `parseId` if you ever intend to support ID 0 (you don't, so that one is fine as-is).

---

### 2. `Content-Disposition` not sanitized on legacy file serve path

**File:** `app/api/uploads/[filename]/route.ts`, line 69

The Vercel Blob path (line 51) correctly sanitizes `originalName`:

```ts
const safeName = (att.originalName ?? filename).replace(/"/g, "")
```

But the legacy local disk path (line 69) does not:

```ts
"Content-Disposition": `attachment; filename="${att.originalName ?? filename}"`
```

If `originalName` contains a double-quote, it breaks the header. Unlikely to be exploited since only admins upload files, but inconsistent and should match.

**Fix:** Use the same `safeName` variable for both paths.

## 🟡 Duplication (Clean Up)

### 3. `parseDate` is defined twice

`app/actions.ts` and `app/recurring/actions.ts` both contain identical `parseDate` functions. This belongs in `lib/parse.ts` alongside `parseTime`, `parseId`, and `parseReminder`.

### 4. `MIME_MAP` is defined twice

`app/api/upload/route.ts` and `app/api/uploads/[filename]/route.ts` both define the same MIME map. Extract it to `lib/mime.ts` (or just `lib/upload.ts`). When you add a new file type, you currently have to remember to update both files.

### 5. `inputClass` is defined in five places

`lib/styles.ts` exports the canonical `inputClass`, but five other files define their own local version:

- `app/notes/note-item.tsx` — uses `text-sm` and `rounded-md`, differs from canonical `text-base`
- `app/projects/add-project-form.tsx` — uses `rounded-lg` instead of `rounded-md`
- `app/projects/[id]/project-header.tsx` — local copy
- `app/notes/add-note-form.tsx` — local copy
- `app/people-manager.tsx` — uses `py-1.5` and `text-base`, no `w-full`

Some of these differences are intentional (sizing context), some are probably just drift. Either import from `lib/styles.ts` or document that the difference is deliberate with a comment.

## 🟡 Dead Code / Legacy Paths

### 6. Local disk upload path is dead on Vercel

The `removeFile` helper in `app/notes/actions.ts` and the serve route in `app/api/uploads/[filename]/route.ts` both still try to read/delete from `process.cwd()/uploads/`. Vercel's filesystem is ephemeral — any file written there won't survive a cold start. All new uploads go to Vercel Blob (correct).

The legacy path exists for older records that were stored locally before Blob was added. If you've confirmed no local-file attachments remain in the production database, you can delete both legacy branches. If some might still exist, leave them and add a comment explaining why.

## 🟡 Schema Observations

### 7. No indexes on `householdId`

Every main query filters by `householdId` (tasks, people, projects, notes, recurringTasks), but no model has an explicit `@@index([householdId])`. For your personal household with dozens of records, Postgres will never notice. If this ever scales to many households, add them. Not urgent.

### 8. Constrained string fields instead of enums

`role` ("admin" | "member"), `priority` ("high" | "medium" | "low"), `status` ("active" | "paused" | "done"), and `intervalUnit` are all stored as plain `String`. The valid values are enforced at the app layer (well done), but the schema doesn't document the constraint and a direct DB edit could store anything.

Consider converting these to Prisma enums — it's a migration but makes the schema self-documenting and catches issues at the DB level. Not urgent for a personal app but worth doing the next time you're already touching the schema.

## 🟢 Minor Notes

### 9. Hardcoded names in placeholder copy

`app/add-task-form.tsx` has these in `PLACEHOLDERS`:

```ts
"Something for Hudson to do?",
"Quinn's turn to help?",
```

These are your family's names and this is your personal app, so it's fine. Just worth knowing that if this code ever gets shared or used as a template, those need to come out.

### 10. Push `unsubscribe` silently succeeds on server error

In `app/push-manager.tsx`, `unsubscribe()` fires a `DELETE /api/subscribe` but only has `try/finally`, no `catch`. If the server call fails, the UI shows "unsubscribed" and the browser subscription is removed, but the DB record stays. The user won't get notifications (browser subscription gone), but the DB has a stale endpoint. Low real-world impact but easy to improve.

### 11. JWT callback queries the database on every token verification

`lib/auth.ts`'s `jwt` callback hits Prisma on every call to look up `role` and `householdId`. For a personal app with a handful of users this is completely fine. Worth knowing if you ever notice auth-related latency in production.

### 12. Inline server action in `superadmin/page.tsx`

`NewHouseholdForm` defines `handleCreate` as an inline `"use server"` function inside a server component. This works, but it's inconsistent with the rest of the app where server actions live in `actions.ts` files. Minor — either pattern is valid in Next.js, but consistency makes the codebase easier to navigate.

## What's Working Well (Don't Change)

- **Auth guard pattern** — `requireAssignedOrAdmin`, `verifyBelongsToHousehold`, and the household-scoped `where` clauses are consistent and correct throughout.
- **Input validation** — all server actions validate and sanitize inputs before touching the DB. No trust of client state.
- **CSP headers** — solid set of security headers in `next.config.ts`. Google Fonts are self-hosted at build time so `font-src 'self'` is correct.
- **Cron route protection** — `CRON_SECRET` bearer check is the right pattern for Vercel cron jobs.
- **File upload security** — extension allowlist, size double-check (both header and actual file), UUID filenames, and household-scoped access check in the serve route are all correct.
- **TypeScript** — strict mode on, no `any` anywhere in the codebase, Prisma generated types used throughout. Clean.
